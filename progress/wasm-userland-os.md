# WASM を userland にした極小 OS を、ゼロから育てる 📈

> **A living implementation log.** userland が WASM な極小 Linux を、一段ずつ byte 検証しながら建てている。
> 最新の到達を上に積む。Web 開発の語彙で「今どこまで来たか」を訪問者に伝えるのが、この記事の役目。
> (プロジェクトの固有名は、輪郭がもう少し見えてからの命名に残してある。)

---

## 一言でいうと — Web 開発に例えるなら

| このプロジェクトの世界 | Web の世界 |
|---|---|
| WASM アプリ | あなたの Web アプリ |
| WASI(WASM のシステムAPI) | ブラウザ / Node.js が提供する Web API |
| **このランタイム自体** | **その実行環境そのもの** |

React や Express を書いているのではない。**Node.js やブラウザの一部を、自分で実装している。**
最終的に目指すのは「**Node.js なしでも Node.js 向けアプリが動く世界**」——それを OS の地面でやる。

## 土台 — M3「本物の WASI ランタイム」

**標準ツールチェーンが吐いた WASM を、無改造でそのまま実行できるようになった。**

`zig cc -target wasm32-wasi`(= wasi-libc)がコンパイルした C プログラム——`printf`、`malloc`、
浮動小数点、`argc/argv`——が、自作ランタイムの上で走る。しかも成功条件は「動いた」ではない:

> **Node.js の `node:wasi` と V8 を独立した Oracle にして、出力が byte 単位で一致すること。**

「画面が同じ」でも「それらしく動いた」でもなく、**独立実装と最後の 1 byte まで一致した**——それを
成功条件に置いている。理由は後述(「なぜ byte 一致なのか」)。

ランタイムの実体は、**1 ファイルの自作 WASM インタプリタ**。これが同時に PID 1(init)も兼ねる。

## 最新の到達 — コンパイラが吐いた SIMD カーネルが、そのまま走る

**`clang -msimd128` が自動ベクトル化した本物の SIMD カーネルを、自作インタプリタが実行できるようになった。**
もちろん成功条件は変わらない —— 同じ WASM を `node:wasi` にも流し、**出力が byte 単位で一致すること**。

Web の語彙に置き換えるなら: これまで「JavaScript を1行ずつ解釈できる」段階だったものが、
**「ブラウザの SIMD 拡張(WebAssembly SIMD)まで解釈できる」段階に上がった**、ということ。
デモ用に手で書いた命令ではなく、**コンパイラが最適化の結果として勝手に生成したベクトル命令**が通る。

### 設計の核 — 128 bit を、1 slot の機械にどう流すか

このインタプリタは「**1つの値 = 64 bit の1枠**」という前提で作ってある。そこに 128 bit の `v128` を通したい。

最初の実装は素直に **128 bit を 64 bit ×2 枠に割った**。これは stack の上だけで完結する演算では動く。
が、**ローカル変数に置いた瞬間に壊れた** —— ローカルは「1変数 = 1枠」なので、半分しか保持できない。
特にループを跨いで値を持ち回る accumulator で破綻する。
(以前の SIMD デモが動いていたのは、`v128` が一度も変数に落ちず stack 上で完結していたからだった。)

解法は表現の変更だった: **`v128` の実体は VM 内の側テーブルに置き、stack とローカルには「その index」だけを流す。**

```
stack / local に乗る値  →  64 bit の index  →  側テーブル  →  実体の 128 bit
```

こうすると、VM から見て `v128` は **ただの 64 bit 値**になる。結果 `local.get` / `local.set` / `drop` /
`select` / stack の push・pop —— **既存機構を一切変更せずに** 128 bit が通った。
この一手が、実カーネル実行を成立させた中心。

### 実装と、捕まえた罠

- **SIMD 命令を約 25 個実装**(load/store・lane load/store・`splat`・`extract_lane`・`replace_lane`・
  `shuffle`・`eq`・`any_true`・`min_u`・`extend_low_s`・論理演算・`i32x4.add` ほか)。
  実装対象は WABT の `wasm-objdump` で `0xFD` prefix に続く subopcode を実命令名へ解いて特定し、
  `node:wasi` を Oracle に**一命令ずつ**照合した。
- **未実装命令に当たったら clean halt** —— メモリや stack を壊したまま走り続けて、それらしい嘘の答えを
  返すことはしない。
- **実行しない分岐の中の命令も、長さだけは正しく読む必要があった。** SIMD 命令は prefix + 可変長 subopcode +
  即値。命令長を読み違えると `block` / `loop` / `if` / `else` / `end` の対応が崩れ、**実行すらしていない
  命令のせいで制御フローが壊れる**。デコーダ側の走査を直して解決。
- **定数畳み込みで検証が空振りする罠。** テスト用 WASM をそのまま書くと、コンパイラが計算結果を先に畳んで
  SIMD 命令ごと消してしまう。入力を volatile 由来にして、**検証対象の命令が実行時に必ず通る**ことを保証した。

通った実カーネル: `xorsum`(メモリの窓を読み、`v128` accumulator をループを跨いで持ち回る)と、
**clang が自動生成した SIMD 版 popcount**(ニブル単位の lookup table + `shuffle` + `min_u` + 幅拡張 + 縮約)。
単一命令の動作確認ではなく、**コンパイラが組み立てた実用的な SIMD パス**が丸ごと通ったことが要点。

### 計器が間違っていたのは、計器の方ではなかった

このプロジェクトは実行した**命令数(icount)**を性能の物差しにしている。SIMD を足した直後、
ベンチマークで奇妙な結果が出た —— **SIMD 版が、ビット演算で並列化した SWAR 版に負けた**。

| popcount の実装 | 命令数 |
|---|---:|
| SWAR(64bit のビット技) | 207,184 |
| SIMD(`-msimd128`) | 750,144 |
| 素朴な実装 | 879,140 |

最初の解釈は「計器が SIMD の処理幅を評価していないからだ。レーン数で重み付けした第二の採点系を足そう」。
**実装に着手して、この前提が誤りだと分かった。** icount が数えているのは VM が dispatch した WASM 命令数で、
`v128` 命令1つは既に1命令として数えられている —— **16 byte を一度に処理すること自体が、scalar なら
複数命令かかる仕事の圧縮としてそのまま命令数に現れている**。ここへレーン数を掛けたら、SIMD の利得を
**二重計上**することになる。だから重み付けの追加は中止した。

代わりに、I/O と setup のノイズを排して計算が支配的になる **XOR-fold ベンチマーク**を設計した:

| XOR-fold | 命令数 | `node:wasi` との一致 |
|---|---:|---|
| scalar | 2,401,678 | 一致 |
| `v128` | 305,302 | 一致 |

**約 7.9 倍少ない。** つまり計器は SIMD に不公平ではなかった。popcount が負けたのは計器の欠陥ではなく、
**入力が小さくて SIMD の準備コストを回収できなかった**という測定結果として正しい。

> 結論の修正: **SIMD を報いる特別な採点は不要。** 計算律速の条件では、今の icount が SIMD の
> 命令圧縮効果を既に正しく映している。

この一段の価値は「新機能を足したこと」ではなく、**自分の仮説が間違っていると気づいて、実装を止め、
別の実験で計器の妥当性を確かめ直したこと**の方にある。

### その先 — ライフゲームで見る、四段の梯子

計器の妥当性が取れたので、より大きな実ワークロードへ。題材は Conway のライフゲーム(同じ盤面・同じ世代数、
正典の Python 実装と出力 byte 一致):

| 実装 | 命令数 |
|---|---:|
| 素朴 | 36,000,000 |
| 分岐除去などの最適化 | 7,200,000 |
| SWAR(ビット詰め) | 804,000 |
| **真の `v128` SIMD** | **733,000** |

素朴からの差は **約 49 倍**。1セル=1 byte で `v128` = 16 セル並列、水平の隣接読みは byte オフセットの
unaligned load(ビット詰め表現だと 128bit シフトの桁上がり処理が要る、それを回避)、近傍和は 8 方向の
`i8x16.add` —— 愚直 → 分岐除去 → SWAR → 真の SIMD が、同じ問題の上に一列に並んだ。

### 正直な境界 — ここはまだ一般解ではない

- **側テーブルは有限の ring。** 現在 `2^18` entries。生存中の `v128` を保持したまま 262,144 個を超える
  新しい `v128` を割り当てると、古い entry が再利用され、**生存中の index が指す実体が上書きされうる**。
  有界なカーネル(同時生存が少ない)では安全だが、巨大・長時間の SIMD プログラムに対する一般解ではない。
  根本解の候補は liveness tracking / 参照カウント / GC / 世代付き handle / 型付き 2 枠ローカル / 値表現そのものの拡張。
  **今は「有界前提」という制約として明記する**立場を取る。
- **未実装の SIMD 命令が残っている**(signed min、飽和加算、一部の浮動小数 SIMD など —— 現行ワークロードが
  要求していないもの)。到達したら clean halt。
- **命令数は wall-clock ではない。** icount が妥当だと確かめられたのは「命令圧縮の観測」についてであって、
  実時間・キャッシュ挙動・ホスト側実装コストまで説明するものではない。

## 考え方 — OS は「外界との窓」を一枚ずつ増やして育つ

OS は一気に完成しない。**外の世界と通信する窓(システムAPI)を、一つずつ足していく**。窓が増えるほど、
動かせるアプリが増える。これは Web も同じ構造だ:

```
Web:   fetch() / localStorage / console.log() / setTimeout()   ← ブラウザとの窓
WASI:  fd_write / fd_read     / clock_time_get / proc_exit      ← OS との窓
```

host 関数を増やすとは、**外界との境界を、自分の手で一枚ずつ作っていく**こと。
`fd_write` が最初の窓、`fd_read` が双方向の窓、`clock_time_get` が時間への窓、そしていつか QEMU が
ハードウェアへの窓になる。単なる WASI 実装ではなく、**世界との接点を少しずつ増やしていく記録**だ。

### 今 開いている窓 — 標準 WASI 9 関数

`fd_write`(標準出力) · `fd_read`(標準入力) · `clock_time_get`(現在時刻) · `proc_exit`(終了) ·
`fd_fdstat_get` · `fd_seek` · `fd_close` · `args_get` · `args_sizes_get`

直近で **双方向の入出力(stdin)** と **時間** という二つの能力が加わった。

## なぜ「byte 一致」なのか

「画面が同じだから正しい」——これを信じない。同じに見えて中身がずれている失敗は、後で必ず高くつく。

```
node:wasi    → WASM → stdout
自作ランタイム → WASM → stdout
```

この二つの stdout が **完全に同じ byte** なら、「同じ実装」と言える。**画面ではなく byte を信じる。**

この規律には、実際に牙があった:

- **自作の off-by-one を捕まえた。** WASM 命令表を 1 個ずらして実装したバグ(シフト命令の取り違え)は、
  画面上は「それらしく」動いていた。全メモリを V8 と byte 比較して初めて、隠れ層が全ゼロだと分かった。
- **Oracle は独立実装でなければ無意味、と学んだ。** 切り分けの途中で自作の JS 参照インタプリタを Oracle に
  したら、*同じ頭から同じ誤表を写して、同じ場所で同じく壊れた*。だから他人の実装(node:wasi / V8)を頑なに使う。
- **非決定すら byte-verify に載せた。** 時刻は本質的に非決定的で、生の値は一致しない。そこで時計から
  **決定的な述語**——「単調増加か(t2 ≥ t1)」「実時刻が 2020 年以降か」——を導いて byte 比較した。
  壊れた時計(常に 0 を返す等)はこの述語で偽になり、ちゃんと捕まる。非決定を、決定的な不変条件に射影する。

### 正直な境界

byte-verify が保証するのは **実際に実行された経路**だけだ。まだ実行されていない命令のプロング(まれな
浮動小数の丸め、未使用の変換など)は、実装済みでも **未証明の面**として残る。「libc が丸ごと動く」とは
言わない——「**libc のこの経路が byte で動く**」と言う。厳密さこそが、このプロジェクトの通貨だ。

## ここまで登ってきた道

- **M1** — 最初の Hello World が動く。Node.js と byte 一致。
- **M2** — WASM 命令を増やし、実計算・ループ・浮動小数点が動く。Neural Cellular Automata(「種」)が実行可能に。
- **M2.5** — Docker 上で自作 init を **PID 1** として起動。「OS っぽい身体」を持ち始める。
- **M3** — 本物の WASI ランタイム。C / Rust / Zig が吐く標準 wasi-libc プログラムがそのまま動く。
- **SIMD(現在)** — コンパイラが自動ベクトル化した実 SIMD カーネルが、`node:wasi` と byte 一致で走る
  (→ 冒頭「最新の到達」)。

## 今 動くもの(すべて独立 Oracle と byte 一致)

- **基本**: Hello World / `printf` / 数値フォーマット / 浮動小数点演算 / `sqrt`
- **標準ライブラリ**: `argc` · `argv` / `malloc` / `strcmp`
- **アルゴリズム**: 再帰(fib)/ 素数生成(ふるい)/ 文字列操作(反転)
- **複数言語**: C / Rust / Zig —— *同じ一つのランタイム*の上で(userland は言語非依存)
- **Neural CA**: ニューラル・セル・オートマトンを WASM プログラムとして実行
- **initramfs**: 手書きで生成した initramfs を、Node.js が byte 検証
- **SIMD 実カーネル**: `xorsum` / clang 自動生成の SIMD popcount / `v128` 版ライフゲーム

## 二つの身体 — カーネルとの距離

同じ `init`(PID 1)を、カーネルとの距離が違う二つの身体で起こす。実体は 1 ファイルで、
違いは syscall 層(`write` / `read`)のアーキ分岐だけ。インタプリタは共通だ。

| 身体 | 場所 | カーネル | 状態 |
|---|---|---|---|
| **docker** | cloud | host のを *借りる*(x86_64) | 起動・byte 検証済み |
| **QEMU** | local | 最小 Linux を *自分で起こす*(aarch64) | initramfs まで用意、boot は次(First Boot) |
| unikernel | 憧れ | カーネルの下まで手を伸ばす | 偵察のみ |

## 次に開く窓

1. **Neural CA を WASI アプリ化(おすすめ)** — `fd_write` で画像(PPM)を吐く種にする → `docker run`
   だけで画像が出力される。「OS の上で画像生成アプリが動く」という、目に見える成果。
2. **`environ_*`** — 環境変数 API。`getenv()` が動く。実装は小さい。
3. **Self-hosting(Lua)** — Lua インタプリタを *このランタイムの上で* 動かす。「プログラムの上で
   別のプログラムが動く」一段上の世界。Unix・LLVM・Rust・Go がみな通った道だ。

### その先

- **First Boot** — Docker を卒業し、QEMU 上で自作カーネル + 自作 init だけで起動する。「本当に OS として
  起動した」最初の瞬間。
- **OCI** — Docker に頼らず OCI イメージを自前生成。より OS に近い世界へ。

### SIMD の続き(冒頭「最新の到達」から)

- **メモリのピーク計測** — 命令数に加えて実行中のメモリ使用量を測る。側テーブル方式が VM 内部で
  払っているコストを観測するために要る。
- **側テーブルの生存期間管理** — ring 上書きが実際に破綻する条件を合成ワークロードで再現し、
  「有界前提」の境界をテストとして固定する。一般解への置き換えは、その後でいい。

## 最終的な方向 — WASM を一級市民とした OS

今は `WASM → WASI → Host(OS)`。将来は OS そのものを薄くし、WASM を中心に据える:

```
WASM → 自作ランタイム → Linux syscall      あるいは      WASM → 自作ランタイム → Unikernel
```

外部の研究にも同じ地面を掘っている先行例がある(**WALI** = WebAssembly Linux Interface、
WASM から Linux syscall を薄く直叩きする仕様)。この実装の `env.print → write` は、その赤ちゃん版だった。
方向は間違っていなかった、と外が独立に証明してくれた。
