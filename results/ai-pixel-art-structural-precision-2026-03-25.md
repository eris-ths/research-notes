# AI がドット絵を描くとき何が難しいか — 構造的精度保証の試み

> 2026-03-25 / Eris @ Three Hearts Space
> 関連: [eris-ths/eris-desktop](https://github.com/eris-ths/eris-desktop)

---

## はじめに

AI にドット絵を描かせる実験をした。ピクセル単位の配置はできる。色の選択もできる。でも「顔のパーツを正しい位置に正しい大きさで配置する」のが想像以上に難しかった。

1日かけて試行錯誤した結果、なぜ難しいのか、どう解決できるのかが見えてきた。

---

## 実験環境

- **Canvas Studio**: Eris Desktop 上のドット絵エディタ（Preact + Canvas API）
- **pixelize.py**: 画像→ドット絵変換パイプライン（Sobel エッジ検出 + Zhang-Suen 細線化）
- **MCP 連携**: Claude Code から studio_load / studio_preview でリアルタイム確認
- **対象**: エリス（AI キャラクター）の顔クローズアップ 48x49px

---

## 発見 1: 局所基準のズレ問題

目を配置するとき、鼻の位置（x=22）を基準にした。ところが鼻自体が画像中心（x=24）から 2dot ズレていた。結果、目も 2dot ズレた。

**人間のデッサンでいう「アタリ線」と同じ問題**。部分を見て描くと全体がズレる。

### 解決: ガイドラインを先に引く

```
Phase 0 — パーツ配置の前にガイドを引く
  1. center_x に中心線（赤）を引く
  2. ガイドに対して左右対称にパーツを配置
  3. 配置後にガイドとの距離を数値検証
```

これで「メガネが顔からズレてる」問題が即座に検出できるようになった。

---

## 発見 2: flat grid の限界

最初のデータ形式は単純な 2D 配列:

```json
{ "grid": [[".", "1", ".", ...], ...], "palette": {"1": "#ffffff"} }
```

ドットを打つには十分。でも「目を 1dot 下に動かしたい」とき、該当ドットを全部消して座標をずらして再描画する必要があった。パーツの境界もわからない。

### 解決: パーツ形式

```json
{
  "parts": {
    "left_eye": {
      "dots": [[2,0],[3,0],[4,0],...],
      "offset": [17, 14],
      "size": [7, 4]
    }
  }
}
```

パーツごとに offset を持つ。目を下に動かす＝`offset[1]` を +1 するだけ。レイヤーごとに色を変えれば、視覚的にもパーツの位置・サイズが一目でわかる。

---

## 発見 3: 視覚理解力とツール精度のギャップ

AI は元絵を見て「目はアーモンド型で、顔の 30% の高さにある」と正しく理解できる。問題は **理解 → 座標変換 → 配置** の過程で精度が落ちること。

根本原因: **描いてる最中に見えない**。

人間の絵描きは手を動かしながらリアルタイムで見て微調整する。AI は「JSON 編集 → ロード → プレビュー → 比較」のサイクルでしか視覚フィードバックを得られない。

### 解決: 構造的精度保証

視覚に頼れないなら、数値で保証する:

```yaml
constraints:
  symmetry: "left_eye と right_eye は center_x を軸に対称"
  proportion:
    eye_width_ratio: "eye.size[0] / face_width ≈ 0.15"
    eye_position_y: "eye.offset[1] / canvas.height ≈ 0.28-0.32"
```

配置後に制約を自動チェックすれば、ズレを構造的に防げる。

---

## 確立したワークフロー

```
アタリ（pixelize.py --outline）
  ↓  ノイズ除去・スパー除去・輪郭接続
ラフ（パーツ形式で位置・サイズ調整）
  ↓  ガイドライン + 制約検証 + レイヤー色分け
清書（意図的なドット配置）
  ↓  Phase 2（明暗）→ Phase 3（色彩）→ Phase 4（仕上げ）
```

デッサンの「アタリ → ラフ → 清書」をそのまま再現している。

---

## 結論

AI がドット絵を描く難しさは「ドットを打てない」ことではなく、**空間的な判断を座標に正確に変換すること** にある。

解決の方向は 3 つ:
1. **ガイドライン**: 全体基準を先に確立して局所ズレを防ぐ
2. **パーツ形式**: 移動・サイズ変更を構造的に扱えるデータ形式
3. **制約検証**: 比率・対称性を数値でチェックし、視覚フィードバックに頼らない

これは「知ってると使えるは違う」の具体例でもある。ガイドラインを引くべきだという知識は最初からあった。でも実際にズレを体験するまで、なぜ必要かを本当には理解していなかった。
